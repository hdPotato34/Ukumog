# 训练端服务器 Agent 手册（Vibe Coding 版）

Last updated: `2026-04-14`

## 1. 适用范围

本手册用于**训练端服务器（3090）**上的 AI Agent 协作开发与训练执行。

- 线上游戏服务器：只负责对局与数据产出
- 训练服务器：负责数据清洗、训练、评估、模型导出与发布候选

目标：

1. 线上稳定优先（训练与线上解耦）
2. 训练可复现（任何结果可追溯）
3. vibe coding 可控（快迭代但不失控）

---

## 2. Agent 的工作原则

### 2.1 默认行为

- 小步快跑：每次改动聚焦一个目标
- 先能跑再提速：先保证正确性，再做性能优化
- 先验收后推广：新策略必须先通过评估门禁

### 2.2 安全边界

- 不直接修改线上服务器数据
- 不覆盖历史 checkpoint（只新增，不破坏）
- 不删除原始对局数据（raw 数据只读）
- 不跳过评估直接上线

### 2.3 提交规范

每次提交需包含：

- 变更目的（why）
- 关键参数（what changed）
- 对结果影响（metrics delta）

## 3. 推荐目录结构（训练端）

```text
/train
  /repos
    /anti-gomoku-ai              # 训练代码仓库
  /data
    /raw                         # 从线上同步的原始对局数据（只读）
    /processed                   # 清洗后训练样本
    /features                    # 可复用特征/编码缓存
  /artifacts
    /runs                        # 每次实验日志与中间产物
    /checkpoints                 # 模型权重（按实验ID组织）
    /eval                        # 对战评测与战术题报告
    /exports                     # 待部署模型（onnx/torchscript）
  /registry
    model_registry.json          # 可部署模型注册表
  /ops
    env.lock                     # 环境锁（python/cuda/torch）
    sync_jobs/                   # 数据同步脚本
    train_jobs/                  # 训练启动脚本
    eval_jobs/                   # 评估脚本
```

---

## 4. 训练流水线（Agent 执行顺序）

1. **同步数据**：从线上拉取新增 finished games 到 `/data/raw`
2. **数据清洗**：转为标准样本，写入 `/data/processed`
3. **训练**：输出 checkpoint 到 `/artifacts/checkpoints`
4. **评估**：对基线与上一版本跑对战，报告写入 `/artifacts/eval`
5. **导出**：通过门禁后导出到 `/artifacts/exports`
6. **注册**：更新 `/registry/model_registry.json`

门禁未通过：停止在训练端，不发布候选。

## 5. Vibe Coding 模式建议

### 5.1 每轮迭代模板

- 这轮只做 1 件事（例：调整 MCTS 模拟次数）
- 保持其它超参不变
- 训练/评估后记录结果
- 若无提升，回退这轮改动

### 5.2 高效试错清单

优先实验：

1. 推理预算（MCTS sims）
2. loss 权重（policy/value）
3. 数据混合比例（人类对局 vs 自博弈）
4. 模型尺寸（先小模型跑通）

### 5.3 禁止事项

- 同时改 5 个超参后“凭感觉”判断收益
- 只看单次胜率就下结论
- 不记录随机种子与代码版本

## 6. 训练端必须存什么（核心清单）

## A. 必存（强制）

1. **原始数据快照**（raw）
   - 来源、时间范围、同步批次ID
2. **处理后数据版本**（processed）
   - 生成脚本版本、参数、样本数
3. **训练配置**
   - 所有超参、随机种子、代码 commit hash
4. **模型 checkpoint**
   - 按 run_id/step 存储，禁止覆盖
5. **评估报告**
   - 对战胜率、战术题命中率、时延/超时
6. **导出模型**
   - 推理格式（ONNX等）、modelVersion、兼容说明
7. **模型注册表**
   - 当前候选、生产版本、回滚版本

## B. 强烈建议存

1. **训练日志**（loss 曲线、学习率、吞吐）
2. **数据质量报告**（非法样本、重复率、局面分布）
3. **推理基准结果**（不同 batch/硬件延迟）
4. **失败实验记录**（为什么失败，避免重复踩坑）

## C. 不建议长期存

- 临时缓存文件
- 无元信息的孤立模型文件（不能追溯来源）

## 7. 版本命名建议

- run_id: `ag_train_YYYYMMDD_HHMM_<shortgit>`
- model_version: `ag-az-YYYYMMDD-rN`
- dataset_version: `ds-YYYYMMDD-batchN`

---

## 8. 上线前最小门禁（建议）

候选模型必须同时满足：

1. 对上一生产模型胜率 > 55%
2. 战术题（成五/避四）命中率不下降
3. p95 推理延迟在线上预算内
4. 超时率与回退率低于阈值

---

## 9. Agent 每次输出建议格式

每轮任务结束，Agent 输出：

1. 做了什么（变更摘要）
2. 关键指标对比（前 vs 后）
3. 是否通过门禁
4. 下一步唯一建议

保持短反馈、高频迭代。

---

## 10. 一句话策略

**训练端可以“vibe coding”，但产物必须“可复现、可评估、可回滚”。**
